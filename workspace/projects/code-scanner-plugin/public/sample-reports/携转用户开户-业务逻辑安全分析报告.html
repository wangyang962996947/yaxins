2.1 提交开户流程 - 开户提交接口重复提交（幂等性缺失）

漏洞类型： 业务流程绕过/参数重放

风险等级： 高危

漏洞描述： onTradeSubmit 接口调用 createNpSubscriber 创建用户、账户、订购和携转关系记录，整个流程无任何幂等性控制。攻击者可通过拦截请求并重复发送，在同一携入号码上创建多个用户账户，导致重复开户、资源浪费、计费混乱等严重后果。业务逻辑说明文档 6.5 节已明确标注"重复提交会创建重复用户"。

关键代码片段：

NpSubscriberOpen.java:134-185

public void onTradeSubmit(IRequestCycle cycle) throws Exception
{
    IData param = this.getData();
    IData acctInfo = this.getData("acctinfo");
    param.putAll(acctInfo);
    IDataset offerList = new DatasetList(param.getString("SELECTED_OFFER_VALUE"));
    // ... 组装offerList
    param.put("OFFERS", offerList);
    param.put("ROUTE_CODE",this.getVisit().getLoginEparchyCode());

    IData result  = new DataMap();
    boolean hasError = false;
    try{
        result  =this.call("OrderCentre.person.INpSubscriberOpenOperateSV.createNpSubscriber", param);
    }catch(Exception e) {
        String eMsg = e.getMessage();
        if(!StringUtil.contains(eMsg,"证件号码被预占")){
            hasError=true;
        }
        throw e;
    }finally{
        if(hasError){
            //证件解预占
            IData input = new DataMap();
            input.put("CUST_NAME", param.getString("PARTY_NAME",param.getString("UP_NAME","")));
            input.put("IDEN_TYPE_ID", idenTypeID);
            input.put("IDEN_NR", param.getString("IDEN_NR"));
            input.put("CAMP_ON", "02");
            this.call("CustomerCentre.custmgr.ICCOutOperateSV.cardCampOn", input);
        }
    }
    setAjax(result);
}

修复建议：

  • 代码层面： 在 onTradeSubmit 入口处增加唯一性校验，基于 ACCESS_NUM + 当前日期 + 操作员ID 生成请求指纹，写入 Redis 或数据库去重表，设置合理过期时间（如5分钟）
  • 代码层面： 在 createNpSubscriber 服务中增加前置校验：SELECT COUNT(*) FROM 用户表 WHERE ACCESS_NUM = ? AND STATUS = '有效'，若已存在则拒绝
  • 架构层面： 引入分布式锁（Redis SETNX），以 ACCESS_NUM 为 key，在开户提交期间持有锁，防止并发提交
  • 配置层面： 数据库层面对 ACCESS_NUM + STATUS 添加唯一约束

参考： 类似案例 - CWE-362: Concurrent Execution using Shared Resource with Improper Synchronization

2.2 携入号码校验流程 - 归属地校验硬编码绕过

漏洞类型： 权限与访问控制/权限提升路径

2.4 提交开户流程 - NP_VERIFY记录删除与开户操作非原子性

漏洞类型： 业务流程绕过/状态机异常

风险等级： 高危

漏洞描述： 在 CreateNpDistrictVerb.run() 中，NP_VERIFY（携入申请审核）记录在开户流程中间阶段被物理删除。如果后续步骤（资源选占、账户创建等）失败，NP_VERIFY 记录已被删除但开户未完成，导致该携入号码无法再次办理开户（查询不到审核记录），形成业务死锁。

2.7 提交开户流程 - 随机密码弱随机数生成

        temp.append(a);
    }
    return temp.toString();
}
