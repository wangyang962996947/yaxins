## 一、为什么需要 Harness 工程

### Demo 与生产环境的差异

| | Demo 环境 | 生产环境 |
|---|---|---|
| 输入 | 可控 | 不可控 |
| 场景 | 有限 | 无限 |
| 对抗 | 无 | 有 |
| 稳定性 | 无要求 | 每次调用都要求正确 |
| 错误后果 | 无 | 有且成本有约束 |

**关键认知：差距不在模型，而在工程化能力。**

Anthropic 的实验表明：同一个模型，有 Harness 和没有 Harness 时，结果差异巨大。

### Harness 的本质

Harness 英文原意是"马具"——它的工程思想是：**确保模型能力被正确、稳定、可控地释放**，就像给大模型配上好的马具，让其力量得到有效控制。

## 二、Harness 工程四层架构

```
用户输入
   ↓
┌─────────────┐  不符合 → 拒绝/转人工
│   约束层     │  符合 ↓
│ (系统边界)   │
└─────────────┘
   ↓
┌─────────────┐
│   信息层     │  提供上下文信息（RAG、历史管理、知识注入）
│ (上下文工程)  │
└─────────────┘
   ↓
┌─────────────┐
│   模型推理   │
└─────────────┘
   ↓
┌─────────────┐  失败 → ┌─────────────┐
│   验证层     │ ─────→ │   修正层     │
│ (质量检验)   │        │ (保底机制)   │
└─────────────┘        └─────────────┘
   ↓ 通过
  用户
```

### 1. 约束层 — 定义系统边界

**核心功能：** 定义 AI 能做什么、不能做什么、做成什么样。

**三类约束：**
- **角色约束**：规定 AI 的身份（如"亚信科技客服助手"）
- **输出格式约束**：规定输出必须是 JSON/某格式
- **安全约束**：不能透露系统提示内容等

**常见坑：**
- 写得太宽 → 形同虚设
- 写得太窄 → 影响用户体验

> 设计要点：度的把握需在实际工程中反复调试。

### 2. 信息层 — 提供任务所需的上下文

**核心功能：** 为模型提供完成任务所需的上下文信息。

**典型技术：**
- RAG 检索注入（用户套餐信息等）
- 多轮会话管理的裁剪和压缩策略
- 业务知识知识库注入

**关键原则：**

> 信息要**刚好够且精准**——过多浪费 TOKEN 且引入噪音，过少则回答不准确。

### 3. 验证层 — 输出质量检验

**核心功能：** 在模型输出到达用户之前进行质量检验。

**四类检查：**
- **格式检查**：是否合法 JSON
- **事实验证**：账单金额是否与数据库一致
- **安全验证**：是否包含敏感信息
- **一致性验证**：答案是否自相矛盾

**验证失败决策策略：**
- 直接拒绝回答
- 降级到安全兜底回答
- 转人工处理

### 4. 修正层 — 保底机制

**典型策略：**
- **重试**：调整参数重新生成
- **降级**：返回预设安全答案
- **人工兜底**：置信度低于阈值转接人工
- **反馈闭环**：收集触发修正的案例优化前三层

**关键原则：**

> 修正层是保底机制，**不应频繁触发**。若频繁触发，说明前三层存在问题。

## 三、各工程间的关系

| 工程 | 解决的问题 | 作用域 |
|---|---|---|
| **提示工程** | 单次调用中如何写好 prompt | 一次 API 调用 |
| **上下文工程** | 单次调用中如何给模型足够且准确的信息 | 一次或一轮调用 |
| **Harness 工程** | 整个 AI 系统如何做稳 | 完整输入-输出链路 |

**Harness 工程 = 提示工程 + 上下文工程 + 验证机制 + 修正机制 + 系统级安全约束**

## 四、实践案例

### 案例 1：Anthropic 实验对比

| 层级 | 无 Harness | 有 Harness |
|---|---|---|
| 约束层 | 无约束，agent 随意操作 | planner 形成包含 16 个功能的产品规格 |
| 信息层 | 上下文窗口易被占满，早期设计易被遗忘 | 有上下文管理策略，agent 知道已做和待做 |
| 验证层 | agent 自我感觉良好，实际输出质量差 | 独立 evaluator agent，发现代码隐藏 bug |
| 修正层 | — | 发现问题后与 Generator 协商修正，多轮仍有问题则跳过保证进度 |

**结论：** 加 4 层 Harness 后产出改善，虽成本提高 20 倍，但让产品从不可用到可用。

### 案例 2：产品报价管理系统

- **约束层**：编写宪法文件，规定技术栈边界（禁止使用 any 类型、禁止自定义 CSS 等）
- **信息层**：每个 Sprint 开始时注入 spec、plan 和 progress 文件
- **验证层**：设计独立 evaluator 角色，严格测试纪律（四项评估维度全部通过才认定 pass）
- **修正层**：输出分级问题清单，Generator 修复后再验证

### 案例 3：OpenAI Codex CLI

- **约束层**："宁可少做、不要多做"，绝不使用破坏性命令，不主动提交代码
- **信息层**：设计 agents Markdown 文档机制，分层注入信息
- **验证层**：由窄到宽的验证策略，先跑最相关测试，再扩大范围
- **修正层**：格式修复最多重试三次，三次后放弃格式修复，提交功能正确代码并说明问题

## 五、用 Harness 四层诊断 AI 系统

拿到一个 AI 项目，可以用四层框架来诊断：

| 层级 | 诊断问题 |
|---|---|
| 约束层 | 有无明确角色边界？约束是否清晰可执行？ |
| 信息层 | 模型能否获取业务数据？注入方式是什么？ |
| 验证层 | 有无质量检查？检查哪些维度？ |
| 修正层 | 有无重试机制？失败后如何处理？ |

## 七、社区最佳实践（来自 GitHub 开源社区总结）

### 架构原则

1. **多代理模式，职责分离**：将工作分配给专门代理，而不是让单一代理过载。典型分工：
   - **规划师 (Planner)**：将简短提示扩展为完整产品/功能规格，聚焦产品上下文和高级技术设计，而非颗粒化实现细节
   - **生成器 (Generator/Builder)**：按规格一次实现一个功能，增量开发
   - **评估者 (Evaluator/QA)**：像用户一样测试运行应用，根据标准对输出评分，并向生成器提供具体反馈进行迭代

2. **保持单体直到复杂要求拆分**：在达到单一代理无法克服的特定瓶颈之前，先从单代理循环开始，不要过早引入多代理编排

3. **每个会话重置上下文**：为每个会话创建新客户端/上下文，而不是仅依赖压缩。完整重置提供干净的状态，结构化工件衔接间隙

### 状态管理与持久化

- **状态持久化到结构化文件，而不是上下文**：任务列表使用 JSON（比自由格式 Markdown 更抗模型诱导损坏）、进度笔记、计划/规格文件、初始化脚本都需要写入磁盘
- **使用仓库作为记录系统**：将所有相关知识推送到仓库，不能被代理发现的知识不存在
- **AGENTS.md / CLAUDE.md 应该是地图，而不是百科全书**：顶级指令文件保持简短（~100行），用作目录指向 `docs/` 目录中更深层的真实来源，渐进式披露知识

### 会话协议

- 每个会话遵循可预测的生命周期：定向 → 设置 → 验证基线 → 选择一个任务 → 实现 → 测试 → 更新状态 → 干净退出
- **每个会话只处理一个任务**：防止上下文耗尽，保持每个会话专注和可恢复
- **构建前先验证**：会话开始时始终运行基线验证，跨会话复合错误是最常见的失败模式之一

### 反馈循环与反压

- **自动化验证作为反压**：类型检查器、linters、测试套件、静态分析器、安全扫描器，关键约束：反馈轮必须快
- **使用浏览器/UI 自动化进行验证**：强制代理与运行中的应用程序交互，捕获后端测试遗漏的错误
- **构建具有具体评分标准的评估器**：不要问"好不好"，要定义具体的、可评分标准
- **复杂工作使用 sprint 合同**：在每个实现块之前，生成器和评估者协商"完成"是什么样子

### 上下文窗口管理

- **将上下文视为稀缺资源**：将昂贵工作卸载给子代理，确定性地每次循环加载相同核心文件，指导代理在实现前搜索以避免重复实现
- **战略性使用子代理**：只读操作（搜索、分析）扇出并行，写操作（构建、测试）限制并行性避免反压问题，让子代理汇总结果而不是将原始输出转储到主上下文

### 安全与沙箱

深度防御三层：
1. **OS 级沙箱** — 隔离代理执行环境
2. **文件系统限制** — 将文件操作限制在项目目录
3. **命令允许列表** — 只允许代理需要的命令，对敏感命令添加验证

### 代码质量与架构

- **机械地强制执行不变量，而不是仅通过文档**：使用 linters、结构测试和 CI 检查来强制执行架构规则，自定义 lint 错误消息应包含修复说明
- **技术债务视为垃圾收集**：运行定期清理代理扫描偏离"黄金原则"的地方，以小增量持续偿还债务，而不是让它复利
- **优化代理可读性，而不仅仅是人类可读性**：支持"无聊"技术（可组合、稳定 API，训练数据中充分表示），而不是难以让代理建模的前沿技术
- **让应用程序可被代理检查**：每个 git worktree 启动应用程序以便代理可以运行隔离实例，将浏览器 DevTools 协议连接到代理运行时，通过可查询 API 公开日志、指标和追踪

### 恢复能力

- **Git 是你的安全网**：每个成功任务后使用描述性消息提交，会话开始时读取最近 git 历史记录，使用 git 标签标记已知良好状态，代理生成损坏代码库时 `git reset --hard` 到最后一个良好状态重新运行
- **预期失败并为之计划**：你会醒来看到损坏的构建，问题总是：重置重新运行更容易，还是制作提示拯救当前状态更容易？两种策略都有效
- **定期重新生成计划**：待办事项列表和计划会随时间漂移，定期让代理将当前代码库与规范进行比较，删除并重新生成它们，防止代理遵循陈旧或不正确的计划

## 八、关键语录

> "AI 应用从 Demo 到生产，关键不在模型，而在 Harness 工程。"

> "修正层是保底机制，若频繁触发，说明前三层存在问题。"

> "信息层提供信息要刚好够且精准，过多浪费 TOKEN 且引入噪音，过少则回答不准确。"

> **核心原则总结（来自社区）**：
> 1. 上下文窗口是约束；结构化工件是解决方案
> 2. 分离生成与评估 — 代理无法客观判断自己的工作
> 3. 每个会话一个任务 — 专注防止上下文耗尽，保持会话可恢复
> 4. 构建前验证 — 始终检查上一个会话没有破坏东西
> 5. 接入快速反馈循环 — 测试、linters、类型检查和 UI 自动化作为反压
> 6. 仓库是唯一真实来源 — 如果不在仓库中，对代理就不存在
> 7. 人类掌舵，代理执行 — 工程师设计环境，指定意图，构建反馈循环，代理编写代码
> 8. 期望最终一致性 — 大多数问题可以通过更好调优提示的更多循环解决
> 9. 不断简化 — 剥去模型不再需要的线束复杂性
