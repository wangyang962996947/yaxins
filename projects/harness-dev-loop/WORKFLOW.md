# Harness 工程驱动开发循环

> 用 Harness 四层框架规范：需求 → 开发 → 验证 → 修 Bug → 验证 → 修 Bug 循环

---

## 一、整体架构

```
┌─────────────────────────────────────────────────────────┐
│                      约束层                              │
│  角色定义 | 范围边界 | 完成标准 | 退出条件               │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                      信息层                              │
│  需求上下文 | 代码上下文 | 测试数据 | 已知问题            │
└─────────────────────────────────────────────────────────┘
                            ↓
                    ┌───────────┐
                    │  模型推理  │  ← Builder（生成者）
                    └───────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                      验证层                              │
│  类型检查 | 单元测试 | 集成测试 | 手工验收               │
│              ↑ 独立于 Builder 的 QA 角色                  │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                      修正层                              │
│  重试(≤3次) | 降级 | 转人工兜底                         │
│  修正层频繁触发 → 说明前三层设计有问题                   │
└─────────────────────────────────────────────────────────┘
```

---

## 二、每轮循环的四个阶段

### 阶段 1：需求确认（约束层 + 信息层）

**目标：** 明确要做什么、不做什么、什么叫完成

**产出：**

```markdown
## 本轮需求

**功能/问题：** [一句话描述]

**约束（Must / Should / Could）：**
- Must：必须实现的功能
- Should：建议实现
- Could：可以做但不是必须的

**验收条件（可量化）：**
- [ ] 条件1：具体、可测试
- [ ] 条件2：必须能用自动化测试验证

**退出标准：**
- 所有 Must 条件通过
- 无新增高危 bug
```

**信息注入：**
- 相关代码文件路径
- 技术栈限制
- 已有类似实现参考

---

### 阶段 2：开发实现（Builder 执行）

**原则：** 一次只做一个功能点，不要贪多

**执行步骤：**

1. **约束检查** — 确认范围，不做超纲的事
2. **增量实现** — 按验收条件逐条实现
3. **自测** — 提交前跑本地检查（类型、lint、简单测试）
4. **提交代码** — Git 提交，消息描述本轮做了什么

**Git 提交规范：**
```
[type] 简短描述

- 具体改了什么
- 对应的验收条件
```

---

### 阶段 3：验证（Evaluator 执行）

**原则：** Validator 必须独立于 Builder，不能自己写代码自己测

**验证清单：**

| 验证项 | 工具/方法 | 通过标准 |
|--------|-----------|---------|
| TypeScript 类型 | `tsc --noEmit` | 零错误 |
| Lint | ESLint | 零警告（或已知警告清单） |
| 单元测试 | Vitest/Jest | 覆盖率 ≥ 80%，全部通过 |
| 功能验收 | 手工 + 自动化 | 所有 Must 验收条件通过 |
| 回归测试 | 已有测试套件 | 全部通过，无退化 |

**验证结果记录：**
```markdown
## 验证报告

| 验证项 | 结果 | 详情 |
|--------|------|------|
| 类型检查 | ✅ 通过 / ❌ 失败 | 3 个错误 |
| 单元测试 | ✅ 通过 | 12 个用例全部通过 |
| 功能验收 | ⚠️ 部分通过 | 5/6 条件通过 |

**未通过项：**
1. [ ] 条件X：原因分析
```

---

### 阶段 4：修正（修 Bug 循环）

**进入条件：** 验证失败

**修 Bug 流程：**

```
发现 Bug
   ↓
分析根因（在哪个层出了问题？）
   ├─ 约束层问题？ → 需求/范围定义不清
   ├─ 信息层问题？ → 上下文缺失/噪音太多
   ├─ 验证层问题？ → 测试本身有缺陷
   └─ 实现层问题？ → 代码逻辑错误
   ↓
修正对应层
   ↓
重新验证（回到阶段3）
   ↓
通过 → 进入下一循环
失败 → 修正（最多3次）→ 超过3次降级处理
```

**修正层策略：**

| 策略 | 触发条件 | 操作 |
|------|---------|------|
| **重试** | 首次失败 | 修正代码，重新验证 |
| **分析** | 二次失败 | 定位根因，判断是否前三层问题 |
| **降级** | 三次失败 | 降级 feature 或延后处理 |
| **转人工** | AI 无法解决 | 通知人工处理，记录问题 |

## 三、完整循环示例

以"代码扫描插件的 ZIP 上传功能"为例：

### 第一轮：需求确认

```
需求：用户能上传 ZIP 文件到代码扫描插件

约束：
- Must：支持 .zip 格式，大小限制 100MB
- Should：显示上传进度
- Could：支持拖拽上传

验收条件：
- [ ] 上传 ZIP 后显示文件名
- [ ] 文件类型非 ZIP 时提示错误
- [ ] 上传后进入下一步骤
```

### 第二轮：开发实现

```
Builder 实现：
1. 拖拽区域 + 点击上传
2. 文件类型校验
3. 步骤状态切换
4. 本地 lint + 类型检查通过
```

### 第三轮：验证

```
Validator 验证：
- [x] 类型检查：tsc --noEmit ✅
- [x] 上传 ZIP 显示文件名 ✅
- [x] 非 ZIP 文件提示错误 ✅
- [x] 步骤切换 ✅
- [ ] 100MB 限制 ⚠️ 未实现

未通过：需要加文件大小校验
```

### 第四轮：修正（第一轮修 Bug）

```
Bug：未实现 100MB 文件大小限制

分析：约束层写的是"大小限制 100MB"，但实现时漏了
修正：增加文件大小校验，超过 100MB 拒绝
重新验证：通过 ✅

本轮完成，进入下一轮
```

## 四、四层框架诊断问题

| 问题表现 | 根因所在层 | 解决方案 |
|---------|-----------|---------|
| AI 总做超纲的事 | 约束层太宽 | 收紧范围，明确禁止项 |
| AI 回答不准确 | 信息层缺失/噪音 | 补充上下文，精简信息 |
| 验证总是失败 | 验证层设计有问题 | 修正验证标准 |
| 修正层频繁触发 | 前三层设计有缺陷 | 回炉重新设计前三层 |
